3.3. JVM Architecture

JVM Architecture

3.3.1. Class Loader

  • There are three phases of class loader:
    1. Loading
    2. Linking
    3. Initialization
  1. Loading

    In this phase, class loader reads the .class file, generate the corresponding binary data and save it in the method area.

    • It stores the following information in the method area:
      • the fully qualified name of the loaded class and its immediate parent class
      • whether the .class file is related to Class or Interface or Enum
      • Modifier, Variables and Method information etc.

    After loading .class file, JVM creates an object of type Class(stores info about a class) to represent this file in the heap. 

    public class Student {
    private String name;
    private int roll_No;
    public String getName() {}
    public void setName() {}
    public int getRoll_no() {}
    public void setRoll_no() {}
}
public class Test {
  public static void main(String[] args)
  {
    Student s1 = new Student();
    Class c1 = s1.getClass(); // c1 is stored in heap by JVM
    System.out.println(c1.getName()); // Student
    Method m[] = c1.getDeclaredMethods();
    for (Method method : m)
    {
      System.out.println(method.getName()); // getName, setName, getRoll_no, setRoll_no
    }
    Field f[] = c1.getDeclaredFields();
    for (Field field : f)
    {
      System.out.println(field.getName()); // name, roll_No
    }
    // For every loaded .class file, only one object of type Class is created.
    Student s2 = new Student();
    Class c2 = s2.getClass();
    assert c1 == c2; // true
  }
}
  2. Linking
    • In this phase, class loader performs three actions:
      • Verification: It checks whether this file is properly formatted and generated by a valid compiler or not. If valid, ready for compilation.
      • Preparation: JVM allocates memory for class variables and initializing the memory to default values.
      • Resolution: replacing references.
  3. Initialization

    All static variables are assigned with their values defined in the code and static block.

    • There are three loaders:
      • Bootstrap class loader It loads trusted classes(e.g. String), loads core Java API in bootstrap path, implementing in native languages like C, C++.
      • Extension class loader It is a child of the bootstrap class loader, loading the classes present in the extension directories, implementing in Java by sun.misc.Launcher$ExtClassLoader
      • System/Application class loader It is a child of the extension class loader(e.g. user-defined class), loading classes from application classpath, implemented in Java by sun.misc.Launcher$AppClassLoader

3.3.2. JVM memory

  • JVM memory contains:
    • Method Area one per JVM, shared resource among all JVM threads, stores per-class structures including class name, immediate parent class name, methods and variables information
    • Heap Area: one per JVM
    • Stack Area: For every thread, JVM creates one run-time stack. Each block inside the stack is a frame, storing method calls.
    • PC Registers It stores address of current execution instruction of a thread, each thread has seperate PC registers.
    • Native Method Stacks per-thread, storing native method information

3.3.3. Execution Engine

  • Execution process:
    1. reads the byte-code line by line
    2. uses data and information present in various memory area
    3. executes instruction
  • Three parts of Execution Engine:
    • Interpreter It interprets the bytecode and executes. CONs: every time a method is called, interpretation is required
    • JIT(Just-In-Time) Compiler It increases the efficiency of an interpreter, compiles the entire bytecode and changes it into native code, so re-interpretation is not required. JVM can directly run the native code.
    • Garbage Collector It destroys un-referenced objects.

3.3.4. JNI(Java Native Interface)

It interacts with the Native Method Libraries and provides the native libraries(C, C++) required for execution.

3.4. Execution of Java

Java is platform-independent but JVM is platform dependent.

6. Strings

6.1. Overview

Strings in Java are Objects that are backed internally by a char array. Since arrays are immutable, Strings are immutable as well. Whenever a change is made to a String, an entirely new String is created.

There are two ways to initialize a String object: 

public class Test {
    public static void main(String[] args) {
        String str1 = "abc"; // literal, put in String Constant Pool for optimization

        String str2 = new String("abc"); // put in heap;
        String str3 = str2.intern(); // use intern to put into String Constant Pool
    }
}

6.1.1. String Constant Pool

When we create a String like String str = "abc";, JVM will first check the String Constant Pool. If the string does not exist, then a new string instance is created and placed in a pool. If exist, it will not create a new object but return the reference to it. String Constant Pool resides in the main heap area after JDK7 for unlimited space.

6.1.2. String is immutable

  • Why String is immutable?
    1. The String Constant Pool cannot be possible if String is mutable. If not mutable, it will pose a serious security threat to the application. Password is passed in String. If mutable, it might be changed.
    2. Safe for multithreading for its immutableness.
    3. Immutability gives the security of loading the correct class by ClassLoader.

6.1.3. Reference count

String s1 = "abc";
String s2 = s1; // one more reference count, no more space.

6.2. String Alternatives

6.2.1. StringBuffer(Recommended)

StringBuffer is a peer class of String that provides much of the functionality of strings. StringBuffer represents growable and writable character sequences. Safe for multithreading

6.2.2. StringBuilder

StringBuilder represents a mutable sequence of characters. An alternative to String, it creates a mutable sequence of characters. Not safe for synchronization

6.2.3. StringTokenizer

StringTokenizer is used to break a string into tokens. A StringTokenizer object internally maintains a current position within the string to be tokenized. A token is returned by taking a substring of the string that was used to create the StringTokenizer object. 

import java.util.StringTokenizer;
public class Test {
    public static void main(String[] args) {
        StringTokenizer st = new StringTokenizer("Hello World from China.");
        System.out.println(st.countTokens());
        int index = 0;
        while (st.hasMoreTokens()) {
            ++index;
            System.out.println("Word " + index + ": " + st.nextToken());
        }
    }
}

6.2.4. StringJoiner

StringJoiner is used to construct a sequence of characters seperated by a delimiter and optionally starting with a supplied prefix or ending with a supplied suffix.

7. Arrays

7.1. Features of Arrays

  1. Arrays are dynamically allocated.
  2. Arrays are objects. So we can find their length using length.
  3. Variables in arrays are ordered.
  4. An array can be used as a static field, local variable or method parameter.
  5. The direct superclass of an array type is Object.
  6. Every array type implements the intefaces Cloneable and java.io.Serializable. (See Marker Interface)
  7. Arrays support primitive types and object types: If it’s an array of primitive types, the actual values are stored in the array. If it’s an array of object types, the references are stored in the array.

7.1.1. Final Arrays

When an object is decorated by final keyword, it cannot refer to another object. If an Array object is final, it cannot be assigned to another Array, but the elements inside can be changed.

7.1.2. Reflection Array

Normal Arrays is of type java.util.Arrays. Reflection Array is of type java.lang.reflect.Array. This Array class provides static methods to create and access Java arrays dynamically. reflect.Array keeps the array to be type-safe. 

import java.lang.reflect.Array;
import java.util.Arrays;

public class Test {
    public static void main(String[] args) {
        int sizeOfArray = 5;
        int[] intArray = (int[])Array.newInstance(int.class, sizeOfArray);
        System.out.println(Arrays.toString(intArray));

        Array.setInt(intArray, 0, 5);
        System.out.println(Arrays.toString(intArray));
        System.out.println(Array.getInt(intArray, 0));
    }
}

7.2. Utilizing an Array

public class Test {
    public static void main(String[] args) {
        // initialization
        int[] intArray = new int[]{ 1, 2, 3, 4, 5 };
        // multidimensional array
        int[][] int2DArray = new int[][]{{1, 2}, {3, 4}};
        // clone of arrays
        // "deep copy" for single-dimensional array
        int[] cloneArray = intArray.clone();
        System.out.println(intArray == cloneArray);
        int[][] clone2DArray = int2DArray.clone();
        System.out.println(int2DArray[0] == clone2DArray[0]);
    }
}

8.4. Inheritance

8.4.1. Types of Inheritance

  • Single Inheritance
  • Multilevel Inheritance
  • Hierarchical Inheritance
  • Multiple Inheritance(Through Interface)
  • Hybrid Inheritance(Through Interface)

8.4.2. Important Facts

  • Every class(except Object), is a subclass of the Object class.
  • A subclass can only have a superclass, but can have multiple super interface.
  • Superclass’s constructor will not be inheritted.

8.4.3. Constructors

Constructors will be called from top to bottom.

8.4.4. Multiple Inheritance

Multiple Inheritance is not supported by class, the compiler cannot decide which method to use. But can use Interface + default to realize. Why multiple inheritance is not supported through a class in Java? Because of ambiguity. If multiple inheritance by interface, and there is methods with the same signature, the compiler will medate the implementation in the subclass.

8.4.5. Interfaces and Inheritance

  • A class can extends another class and implement one or more interfaces.
  • An interface can extend one or more interfaces.

8.4.6. Comparison of Inheritance in C++ and Java

  1. In Java, all classes inherit from the Object class implicitly.
  2. In Java, members of the grandparent class are not directly accessible. In C++, can be accessed by :: operator
  3. In Java, protected fields are also accessible by other classes.
  4. In Java, methods are virtual by default, which means they inherently support dynamic dispatch. Can use final to make methods unvirtual.
  5. Java doesn’t support multiple inheritances.

8.5. Abstraction

8.5.1. Rules for abstraction classes and methods

  • Any class that contains one or more abstract methods must also be declared with an abstract keyword
  • abstract keyword cannot be used with final and static
  • Abstract classes can have non-final fields

8.5.2. Control Abstraction and Data Abstraction

  • Data Abstraction: creating complex data types but giving out only essential operations
  • Control Abstraction: abstraction hiding the implementation

8.6. Encapsulation

Providing a protective shield for data and methods hiding in the unit.

8.7. Polymorphism

Polymorphism is that in which we can perform a task in multiple forms or ways.

  • Two types:
    • Compile-time polymorphism static polymorphism achieved by function overloading
    • Runtime polymorphism dynamic method dispatch achieved by method overriding

8.7.1. Runtime Polymorphism

Runtime Polymorphism only support method polymorphism, not data polymorphism. Subclass will have a copy of superclass’s method inside, choosing method by the type of reference.

10. Methods

10.1. Java is strictly passed by value.

When passing a primitive type, only pass a copy of the primitive type. When passing a object type, pass a reference to the object.

10.2. Method Overriding

10.2.1. Overriding methods can allow more, but not less access than the overriden methods

if the superclass method is protected, the subclass method can be protected or public, but not private.

10.2.2. Final methods cannot be overriden

10.2.3. static methods cannot be overriden

If there is a subclass method with the same signature, it’s called method hiding.

10.2.4. private methods cannot be overriden

10.2.5. The overriding method must have the same return type

10.2.6. Invoking overriden method from sub-class

Use super keyword

10.2.7. Overriding and Exception Handling

  1. If the superclass overriden method does not throw an exception, subclass overriding method can only throw the unchecked exception.
  2. If the superclass overriden method does throw an exception, subclass overriding method can only throw the same or child exception, throwing parent exception will cause compiler error.

11. Interface

Used to achieve total abstraction and support multiple inheritance. A class can implement Comparator Interface to be used to sort a collection.

11.1. Inheritance of Interface

  • Why multiple inheritance is not supported through class but interface? To avoid ambiguity. After Java 8, interfaces also have implementation of methods. If there is ambiguity, the compiler will ask you to implement the method.

11.2. Functional Interfaces

A functional interface is an interface that contains only one abstract method. E.g. Runnable, ActionListener, Comparable

11.2.1. @FunctionalInterface Annotation

@FunctionalInterface is used to ensure that the functional interface can’t have more than one abstract method.

11.3. Marker Interface

Empty interface. E.g. Serializable, Clonable, Remote

11.3.1. Clonable

A class that implements Clonable indicates that it is legal for clone() method to make a field-for-field copy. By convention, classes that implement this interface should override Object.clone() method.

11.3.2. Serializable

It is used to make an object eligible for saving its state into a file.

11.3.3. Remote

Making a class remote indicates the object is stored at one machine, and accessed from another machine.
